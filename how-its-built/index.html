HOW IT'S BUILT

How we’re building it - Agile process

We believe to build a great product, you have to embed that product into the real world as early as possible. Gathering all your software requirements up-front is unreliable and often leads to a product that doesn’t properly solve users’ problems.

Instead, we start with a simple product and learn from feedback of it being used for real work. In this way, we accept changing requirements as inevitable and adapt to better suit project needs. Releasing early and visibly also allows stakeholders to track and guide the development process.

How we’re using agile in NRT

Each phase of NRT sets out a release schedule for that period. We aim to release a new version of NRT and gather feedback at least every two months. The feedback we gain from each release sets the objectives for the next, so we avoid setting the goals for releases ahead of this schedule.

Tools we’re using to build NRT

To manage a project like NRT we have a number of key tools that allow us to build quickly and effectively. Here’s a summary of the tools we are using for NRT; we will have more information on how we use them on our developer blog (coming soon).

  • Google drive - The best way to collaborate on documents with a distributed team.
  • GitHub - The source code for NRT is open-source and shared using GitHub https://github.com/unepwcmc/NRT.
  • Node.js, Backbone.js and CoffeeScript - We use Node.js and Backbone.js to build a super-fast application, which uses CoffeeScript all the way from the client to the server.
  • Trello - Gives us visibility on our tasks, and what stage of completion they are in. Trello.com.
